# 협업 운영 규칙 (GitHub + Obsidian)

## 1) 현재 운영 원칙
- `main`은 **공유 기준 브랜치**로 사용한다.
- 현재는 **직접 커밋/푸시가 가능한 운영 상태**이며, 작은 문서 수정은 빠르게 반영해도 된다.
- 다만 큰 구조 변경이나 구현 리스크가 큰 작업은 여전히 **작업 브랜치 분리**를 권장한다.

### 권장 브랜치 네이밍
- `docs/<topic>` (문서)
- `feat/<topic>` (기능)
- `fix/<topic>` (수정)

## 2) GitHub 작업 규칙
- 작은 문서 정리/링크 수정/회의 반영은 `main`에 바로 커밋 가능
- 구현 범위가 크거나 충돌 가능성이 높은 작업은 브랜치에서 진행 후 머지
- PR은 **필수 규칙이 아니라 선택 도구**로 본다
- PR을 사용할 때는 아래를 포함하면 좋다:
  - 변경 이유
  - 변경 파일 목록
  - 영향 범위(어떤 문서/시스템)

## 3) Obsidian 협업 규칙
- 모든 팀원은 **동일 Vault 루트**를 열 것:
  - `캡스톤_다크호러_덱빌딩오토배틀러`
- 구조 변경 시 먼저 `00_프로젝트_허브.md`를 업데이트한다.
- 회의 문서는 `회의록/` 폴더 아래에 저장한다.
- 제출용 문서는 `제출용/` 폴더에서 관리한다.
- 회의록 파일명은 날짜 중심으로 관리한다.
  - 기본: `YYYY-MM-DD_회의.md`
  - 필요 시: `YYYY-MM-DD_결정사항.md`, `YYYY-MM-DD_액션아이템.md`
- 디스코드는 회의 아카이브 저장소로 쓰지 않는다.
  - 디스코드: 실시간 논의 / 임시 로그 / 회의 재구성용 원본
  - Obsidian + GitHub: 최종 회의록 / 문서 반영 / 버전 관리 기준
- 회의 후 10분 내:
  1) `회의록/`에 회의록 저장
  2) 관련 문서 링크 갱신
  3) 허브/설계 문서 반영
  4) GitHub 기준으로 커밋/푸시

## 4) 충돌 방지 규칙
- 기기별 UI 파일은 추적 제외(`.obsidian/workspace*.json`)
- 같은 파일 동시 편집은 피한다
- 큰 문서 수정 전 "내가 이 파일 수정 시작"을 공유한다
- 파일 이동/이름 변경 후에는 허브와 관련 링크를 바로 점검한다

## 5) 리뷰/검토 원칙
- 문서 변경은 가능하면 **바로 읽히는 수준으로 작게** 나눈다
- 수치 변경(밸런스)은 변경 근거 1줄 필수
- 구조 개편, 시스템 기준 변경, 대량 이동은 최소 1회 팀 확인 권장

## 6) 커밋 메시지 컨벤션
- `docs:` 문서 변경
- `feat:` 기능 추가
- `fix:` 버그/오탈자/링크 수정
- `chore:` 구조 정리/설정 변경

예시:
- `docs: 회의 안건 반영 및 용병소 규칙 명문화`
- `feat: 패널 라우터 상태 전환 초안 추가`
- `chore: 제출용 문서 폴더 분리 및 허브 링크 정리`
