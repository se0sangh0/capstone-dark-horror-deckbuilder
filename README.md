# 캡스톤 — 다크 호러 덱빌딩 오토배틀러

팀 문서/기획 협업용 Obsidian Vault + GitHub 저장소입니다.

## 저장소 목적
- 회의 결정사항과 설계 문서를 단일 소스로 관리
- Obsidian에서 빠르게 작성하고 Git으로 안정적으로 협업
- 구현용 Unity 프로젝트와 기획 문서를 한 저장소에서 함께 관리

## 시작하기
1. 저장소 클론
2. Obsidian에서 **이 폴더 자체**를 Vault로 열기
   - `캡스톤_다크호러_덱빌딩오토배틀러`
3. `00_프로젝트_허브.md`부터 읽고 현재 기준 문서를 확인
4. 작은 문서 작업은 바로 커밋해도 되고, 큰 작업은 브랜치 분리 권장

## 문서 구조
- `00_*` : 허브/운영 규칙/협업 규정/UI 가이드
- `01_*` : 코어 컨셉/코어루프
- `02_*` : 전투/카드/스트레스 시스템
- `03_*` : 동료/맵/직업군
- `04_*` : MVP 범위/테스트
- `05_*` : 회의/로드맵/일정
- `06_*` : 학교 제출용 정리/추후개발 백로그
- `07_*` : 개인 체크리스트/주간 작업 메모
- `회의록/` : 회의 기록
- `제출용/` : 제출 초안/양식 매핑
- `archive/` : 병합되었거나 보관용으로 남긴 문서
- `99_*` : 템플릿

## 작성 규칙 (요약)
1. 구조 변경 시 먼저 `00_프로젝트_허브.md` 링크를 갱신한다.
2. 수치 변경은 변경 이유를 최소 1줄 남긴다.
3. 회의 후 10분 내 회의록 + 관련 문서 링크를 갱신한다.
4. 큰 변경 전에는 팀 채널에 “수정 시작 파일”을 공유한다.
5. 확정 명세와 아이디어 메모를 섞지 않는다.
   - 확정된 내용 → 관련 명세 문서
   - 미확정/후순위 아이디어 → 백로그 문서

## Git 협업 규칙
- 현재는 `main`에 직접 커밋/푸시 가능한 운영 상태
- 다만 큰 구조 변경이나 구현 작업은 브랜치 사용 권장
- 브랜치 네이밍:
  - `docs/<topic>`
  - `feat/<topic>`
  - `fix/<topic>`
- 커밋 컨벤션:
  - `docs:` 문서
  - `feat:` 기능
  - `fix:` 수정
  - `chore:` 정리/설정

## Unity 파일 정책
이 저장소는 **필요 소스만 추적**합니다.

### 추적(올라감)
- `Assets/`
- `Packages/`
- `ProjectSettings/`
- 기획/문서(`*.md`)

### 비추적(안 올라감)
- `Library/`, `Temp/`, `Obj/`, `Build/`, `Logs/`, `UserSettings/`
- IDE 생성 파일(`.vs`, `.idea`, `.vscode`, `*.csproj`, `*.sln` 등)
- Syncthing 로컬 아티팩트(`.stfolder`, `.stversions/` 등)
